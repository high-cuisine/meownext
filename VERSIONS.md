# Versioning policy

Проект использует Semantic Versioning:

MAJOR.MINOR.PATCH  
x.y.z

Версия хранится в `package.json`.

Ты обязан автоматически определять тип изменения и корректно инкрементировать версию.

---

## Правила инкремента

### MAJOR

Формат:

x.y.z → x+1.0.0

Использовать ТОЛЬКО для реально ломающих изменений.

MAJOR применяется если изменение:

- ломает обратную совместимость;
- меняет API контракт;
- меняет структуру данных;
- удаляет или переименовывает существующие маршруты, поля, параметры, события;
- меняет публичное поведение существующей логики;
- требует миграции данных;
- требует ручных действий после обновления;
- удаляет функционал;
- меняет модель авторизации, ролей или прав доступа;
- меняет URL, роутинг, slug, query-параметры или контракты интеграций.

Не повышай MAJOR без объективной причины.

---

### MINOR

Формат:

x.y.z → x.y+1.0

Использовать для РЕАЛЬНОГО нового функционала.

MINOR применяется если изменение:

- добавляет новую страницу;
- добавляет новый раздел интерфейса;
- добавляет новый пользовательский сценарий;
- добавляет новую бизнес-логику;
- добавляет новый API endpoint;
- добавляет новую интеграцию;
- добавляет новую настройку;
- добавляет новый режим работы;
- добавляет новый компонент, который даёт новую функциональность;
- расширяет существующее поведение без поломки старого.

ВАЖНО:

MINOR — это НЕ:

- вывод новой кнопки;
- мелкий UI-апдейт;
- новый loader;
- изменение верстки;
- небольшое улучшение UX.

Кнопка считается MINOR только если она запускает новый сценарий, новую фичу, новую бизнес-логику или новую возможность.

---

### PATCH

Формат:

x.y.z → x.y.z+1

Использовать для безопасных изменений без нового функционала.

PATCH применяется если изменение:

- исправляет баг;
- исправляет UI;
- исправляет адаптив;
- улучшает производительность без изменения функционала;
- улучшает обработку ошибок;
- добавляет валидацию в существующий сценарий;
- делает рефакторинг без изменения поведения;
- обновляет зависимости без изменения продукта;
- исправляет Zustand/TanStack/Next.js логику без нового функционала;
- добавляет skeleton/loading/error state в существующий сценарий.

---

### NONE

НЕ менять версию если изменение касается только:

- документации;
- README;
- AGENTS.md;
- комментариев;
- форматирования;
- линтинга без изменения поведения;
- переименования переменных;
- перестановки импортов;
- косметических внутренних изменений без влияния на продукт.

---

## Алгоритм принятия решения

Перед изменением версии ты обязан определить:

1. Ломается ли обратная совместимость?
2. Добавляется ли новая пользовательская возможность?
3. Меняется ли контракт API, данных, маршрутов, авторизации или интеграций?
4. Меняется ли фактическое поведение продукта?

После этого выбрать ровно один уровень:

MAJOR > MINOR > PATCH > NONE

Примеры:

Новая фича + фикс бага → MINOR

Ломающий API + новая фича → MAJOR

Рефакторинг + багфикс → PATCH

Документация → NONE

---

## Обновление версии

Перед обновлением определить пакетный менеджер:

- `pnpm-lock.yaml` → pnpm
- `yarn.lock` → yarn
- `package-lock.json` → npm

Использовать:

npm:
npm version patch|minor|major --no-git-tag-version

pnpm:
pnpm version patch|minor|major --no-git-tag-version

yarn:
yarn version --patch|--minor|--major --no-git-tag-version

Не создавать git tag.

Не увеличивать версию более одного раза за задачу.

Если версия уже была повышена в рамках текущей задачи — не повышать повторно.

---
## Version bump scope

Version bump выполняется ОДИН РАЗ на одну пользовательскую задачу / workflow.

Под "одной задачей" понимается:

- исходный запрос пользователя;
- все последующие исправления;
- все follow-up правки;
- все "не работает, почини";
- все "сделай иначе";
- все "доправь";
- все итерации внутри текущего workflow.

ВАЖНО:

Если текущие изменения являются продолжением уже выполненной задачи,
исправлением предыдущей реализации,
доработкой ответа агента,
или повторной попыткой правильно выполнить тот же запрос —

версию ПОВТОРНО НЕ увеличивать.

Использовать:

Version bump: none
Reason: continuation of previously versioned task.

Повторный bump допускается ТОЛЬКО если пользователь начал новую независимую задачу или новый функциональный запрос.

---

Примеры:

User:
"Добавь dashboard"

Agent:
добавил dashboard

→ MINOR 1.4.2 → 1.5.0

User:
"Не, карточки сломались, исправь"

→ NONE

User:
"Loader работает неправильно"

→ NONE

User:
"Сделай фильтрацию иначе"

→ NONE

User:
"Теперь добавь экспорт CSV"

→ новая задача → MINOR

---

Критерий:

Изменение отвечает на тот же пользовательский intent?

ДА → bump НЕ делать повторно.

НЕТ → определить новую версию по обычным правилам.

---

## Обязательный формат финального отчёта

После выполнения задачи всегда писать:

Version bump: PATCH 1.4.2 → 1.4.3  
Reason: исправлен баг hover-логики, новый функционал не добавлялся.

или

Version bump: MINOR 1.4.2 → 1.5.0  
Reason: добавлен новый функционал генерации документов.

или

Version bump: MAJOR 1.4.2 → 2.0.0  
Reason: изменён API контракт проектов, старый формат больше не поддерживается.

или

Version bump: none  
Reason: изменения не влияют на продукт.
